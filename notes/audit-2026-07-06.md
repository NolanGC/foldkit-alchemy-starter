# Code quality audit — 2026-07-06

Scope: whole app (frontend foldkit Tier 7, backend Effect/Alchemy workers).
Graded against `reference_repos/foldkit` exemplars (typing-game room page) and the
foldkit-skills architecture/conventions/checklist references.

## Exemplar patterns we should adopt

- `makeRemoteData` (typing-game/client/src/makeRemoteData.ts): `Idle | Loading | Error | Ok`
  schema factory for anything fetched. Room page models the room as `RoomRemoteData.Union`
  and the view renders `Loading` explicitly. **This is the fix for the "No messages yet"
  flash**: our chat Model has `messages: []` from the instant a room connects, so the empty
  state renders during loading. History needs a `Loading | Loaded` union.
- `optionWhen` + `Array.fromOption` (typing-game optionWhen.ts, room/update/update.ts):
  0-or-1 command construction, replaces `...(condition ? [Cmd()] : [])` spreads.
- Grouped no-op message handling: `M.tags({...real handlers})` then one
  `M.tag('CompletedA', 'CompletedB', ..., () => [model, []])` then `M.exhaustive`
  (room/update/update.ts:213-226). Our update lists four separate no-op entries.
- Curried extracted handlers `(model) => (message) => UpdateReturn` for anything
  non-trivial (handleKeyPressed, handleRoomUpdated).
- Keyed content branches on remote-data tag (room/view/view.ts `contentKey`).

## Findings

### Frontend `page/chat.ts`

1. **Flash bug / state modeling**: `messages: S.Array(ChatMessage)` + `hasMoreHistory`
   conflate "not loaded" and "loaded, empty". Replace with
   `history: HistoryLoading | HistoryLoaded({ messages, hasMore })`.
2. Stale-room guard `roomId === model.roomId ? [...] : [model, []]` duplicated across
   6 handlers → extract curried `whenCurrentRoom` helper.
3. `SendMessage` / `RequestHistory` / `RequestOlderHistory` share a ~20-line
   `ChatSocket.get` + room check + `catchTag("ResourceNotAvailable")` scaffold →
   extract one socket-frame helper.
4. `...(Option.isNone(model.maybeZone) ? [GetLocalZone()] : [])` → `optionWhen` +
   `Array.fromOption`.
5. `model.messages[0]` bracket indexing → `Array.head` + `Option.match`
   (repo convention: never bracket-index).
6. Four no-op handlers listed individually → group via `M.tags` + multi-tag no-op.
7. `ChatSocketAcquireError extends Error` → `Data.TaggedError`; then
   `onAcquireError`'s `instanceof` + `"unknown"` sentinel roomId can go
   (acquire's error channel is always `ChatSocketAcquireError`).
8. Subscription `Stream.when(stream, Effect.sync(() => isConnected))` →
   `isConnected ? stream : Stream.empty`; `Effect.sync` around a pure bool is noise.
9. View: `...(hasMoreHistory ? [button...] : [])` spread → element-level
   `hasMore ? loadMoreView() : h.empty` (exemplar style), plus history-union branches keyed.
10. `senderId.slice(0, 5)` magic number → named constant.

### Frontend `main.ts`

11. Top-level view branches (landing `h.main` vs app shell `h.div`) not keyed against
    each other → key the body root on a route-group key.
12. `routeRoomId` maps Home → DEFAULT_ROOM_ID; with landing at Home this prep is
    stale-ish; restructure with the rooms feature.

### Backend

13. `DurableObject.ts:32-48`: `toChatMessage` / `toPersistedChatMessage` hand-map fields
    that the `ChatMessage` schema codec already encodes: `S.DateTimeUtcFromMillis`'s
    encoded form IS epoch millis. Make the persistence wire type
    `typeof ChatMessage.Encoded` (+ roomId) and use `S.decodeSync/encodeSync(ChatMessage)`.
    The helpers the user flagged exist only because the wire struct renames the field.
14. `DurableObject.ts:68-69`: `let cachedHistory` / `let cachedHistoryHasMore` mutable
    closure vars → single `Ref<Option<{ messages, hasMore }>>`; two `let`s can drift.
15. `DurableObject.ts:50-54`: `roomIdFromRequestUrl` falls back to `"general"` — a
    sentinel that silently mis-rooms on a routing bug. Fail loudly / pass roomId explicitly.
16. `DurableObject.ts:160`: `body.length === 0` → `String.isEmpty` per conventions.
17. `ChatPersistenceService.ts persistMessage`: single-yield `Effect.gen` wrapper → plain
    pipeline with `Effect.asVoid`.
18. `ChatPersistenceService.ts getRoomHistory`: `rows.slice(0, limit).map(...).reverse()`
    native chain composing → `pipe(rows, Array.take(limit), Array.map(...), Array.reverse)`.
19. `cursor?: ChatHistoryCursor` optional params at the RPC boundary: acceptable
    (structured-clone boundary; `Option` class instances don't survive it) — add NOTE.

### Tests

20. story/scene tests assert the current flat model; they must evolve with the history
    union + rooms feature (update, don't delete coverage).

## Rooms feature design (user request)

- New `Rooms` drizzle table, seeded `general`, `random`, `feature-requests` via a
  hand-written SQL migration (drizzle-kit custom migration format).
- `ChatPersistenceService` gains `listRooms` (and existence is derivable client-side).
- `ChatService` adds `GET /api/rooms` (JSON list) and refuses the socket upgrade with
  404 when the room is not in the table (defense in depth).
- Frontend: app-level `rooms` RemoteData fetched at init via HTTP Command; nav renders
  from it; entering `/chat/<unknown>` renders a simple "Room not found" view without
  connecting. This reuses the same RemoteData idiom as the flash fix.
