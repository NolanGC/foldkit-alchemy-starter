⏺ Confirmed the root cause. The Vite chat app runs as a standalone 
  vite.createServer (Vite.ts:20) with zero connection to the workerd runtime
  hosting your router. There is no shared service registry between them, so the
  router's CHAT_APP service binding (Service.local({ scriptName })) resolves to
  a workerd service that doesn't exist → 500 with no JS stack trace. Direct hits
   to :1340 work because they bypass the registry entirely.

  This is a local-dev-only limitation: in production both become real deployed
  workers and the service binding would work. But you need dev to work.

  The fix

  Website.Vite can't be a service-binding target in dev (and can't host routing
  logic itself — it has no custom main), so the simplest thing that works
  identically in dev and prod is to have the router forward by URL instead of by
   service binding. You already expose chatApp.url; just hand it to the router
  and use fetch.

  What was wrong and why it "got worse"

  It didn't regress from your changes being wrong — the architecture was fine.
  The trap: Website.Vite runs as a standalone vite.createServer process in local
   dev (Vite.ts:20), completely separate from the workerd runtime that hosts
  your StaticSite router. A workerd service binding (env.CHAT_APP.fetch)
  resolves names through workerd's internal service registry, and the Vite dev
  server was never registered there — so the forward 500'd with no JS stack
  trace (the failure is in the runtime layer, below your code). It worked when
  you hit :1340 directly because that skips the registry.

  It would've worked in production (both become real deployed workers), so this
  was a purely local-dev landmine — the worst kind, since that's where you test.

  The fix (2 small edits)

  - worker/WebsiteRouter.ts: forward by URL (fetch) instead of the CHAT_APP
  service binding.
  - alchemy.run.ts: pass CHAT_APP_URL: chatApp.url.as<string>() instead of
  CHAT_APP: chatApp.

  This works identically in dev and prod (chatApp.url is localhost:1340 in dev,
  the deployed worker URL in prod), needs no service binding, and keeps your
  two-deployable landing-site-in-front-of-chat-app design intact. Typecheck
  passes and your running alchemy dev already hot-picked up both changes.

  One note for production: forwarding by public URL adds one worker-to-worker
  network hop (vs. the internal service mesh). For a starter that's negligible
  and worth the simplicity + dev/prod parity. If you ever want the zero-hop
  service binding in prod specifically, that'd mean branching dev vs. prod logic
   — more complexity than this starter needs. WebSocket chat traffic is
  unaffected either way since the browser connects to ChatService directly, not
  through this router.
